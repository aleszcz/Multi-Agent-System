01
The Single-Agent BottleneckWhy one technician can't commission a whole building

Every developer building with AI agents eventually hits the same wall. It works fine for simple tasks — but the moment your project becomes even slightly complex, things fall apart.

The agent gets overwhelmed. Context bloats. Errors compound. And progress slows to a crawl. The issue isn't the model. It's the architecture.

The two root causes

We expect too much from a single agent, and we don't break problems into simple enough, verifiable tasks. The single-agent one-shot is over — not because models aren't powerful enough, but because the architecture doesn't scale.

Trying to make a single AI agent handle everything is like sending one HVAC technician to commission an entire skyscraper alone — running load calculations, balancing air distribution, testing controls, verifying refrigerant charge, and signing off on safety interlocks all at once. It doesn't scale. It doesn't specialise. And it fails.

04
5 Patterns That Actually WorkStart at the top and work your way down

Over testing dozens of workflows across different AI agents, five patterns have proven consistently effective. Each builds on the last in complexity.

01

The Pre-Commissioning Survey

Run many independent checks in parallel, then select the best readings
Easiest · Start here

Before a commissioning crew touches a building, surveyors fan out across every floor independently — one checking duct routing, one logging nameplate data, one photographing control panels. None waits for the others. At the end, the lead engineer reviews and selects what's usable.

Apply this to AI: dispatch multiple subagents with the same brief, let them generate outputs in parallel, then manually curate the results. Every task is fully independent — no file conflicts, no dependency graphs, no merge logic.

Example: dispatch 5 Codex Spark subagents to generate 10 UI variations each. Cherry-pick the best. Discard the rest.

Requirements

Tasks must be fully independent
No shared files between subagents
Human curates final output
Best for: design exploration, variations, test generation
02

The Commissioning Swarm

All trades on site simultaneously — each owns a separate zone
Parallel execution

On a large commercial job, the air-side tech is balancing VAV boxes on floors 1–5 while the controls engineer is mapping BMS points on floors 6–10, and the refrigeration crew is leak-testing the rooftop plant — all at the same time. Each trade owns a distinct zone and never steps on another's work.

Multiple subagents run simultaneously on clearly partitioned scopes. Good fits: building multiple independent app components, writing tests for different modules, porting pages from one framework to another.

Critical rule

Deeply specific scope per subagent
Individual, verifiable acceptance criteria
Clearly documented inter-system dependencies
No shared files. If two subagents must edit the same file, use a different pattern.
03

Commissioning Phases

Each phase signs off before the next one starts
Large projects

HVAC commissioning on a large project runs in strict phases: mechanical completion must be certified before functional testing starts, and functional testing must pass before integrated systems testing begins. Within each phase, multiple technicians work in parallel on their assigned systems.

Apply this structure to AI projects: break the work into phases where each depends on the verified output of the previous one. Within each phase, any number of subagents can run in parallel. Perfect for full app rebuilds or large refactors.

Phase structure

Phase 1 — Mechanical Completion

Map systems, define acceptance criteria

Phase 2 — Functional Testing (parallel)

Multiple technicians, shared scope brief

Phase 3 — Integrated Systems Test

Cross-system checks, failure mode tests

Phase 4 — Handover & Documentation

O&M manuals, as-built drawings, sign-off

04

The Test-and-Handoff Pipeline

Sequential work orders — state lives in test sheets, not memory
Long-horizon tasks

A refrigeration circuit doesn't get commissioned in one pass. One technician pressure-tests the pipework, signs the leak test sheet, and hands it to the next who pulls vacuum, then hands it to the next who charges refrigerant, then to the final tech who verifies superheat and subcooling. Each station does one bounded job and passes the documented result forward.

In this pattern, subagents operate sequentially. Each does one smaller task, validates it, then hands the workpiece to the next. State lives in files and task queues — not in conversation history. Do not drag unrelated prior context through a single giant thread.

Core principle

Each subagent gets only the context for its one step
Handoff documented in files, not chat history
State lives in files and task queues
Best for: research pipelines, hardware bring-up, multi-step deployments
05

The Independent Witness

Separate the installer from the inspector — always
Always. Layer on top of everything.

In formal commissioning, the technician who installs the system is never the same person who witnesses and signs off the acceptance test. An independent commissioning engineer runs the functional test, records the results, and either accepts or rejects the work. If it fails, the installer gets another pass — the witness never touches the installation.

Apply this as a discipline across every other pattern: separate the subagents that write code from those that verify it. One builder works; one or more verifiers run tests independently. If a verifier flags an issue, the builder is respawned with the specific failure. With near-instant models, adding a verification pass is practically free.

The only important rule

Only one builder writes at a time
Multiple verifiers can witness simultaneously
Use automated tests, screenshots, and deterministic checks
Nothing is accepted without a signed test result
05
Builder vs. VerifierThe correction loop that guarantees quality

This is where multi-agent systems become truly powerful. The Builder and Verifier are independent agents — one creates, one witnesses — and nothing is accepted without a signed test result.

🔨 Builder / Installer
Executes the work
Writes code, configures systems, generates outputs. Focused purely on installation. Does not participate in verification.
⇄
🔍 Commissioning Witness
Validates the result
Runs tests, checks outputs against acceptance criteria, and records results. Does not touch the installation. Flags failures with specific feedback.
Correction loop: Install → Witness test → Fail? Respawn with defect list → Rectify & retest → ✓ Accepted

This is the difference between "it runs" and "it actually works." The sequential loop consists of: one subagent breaking the project into acceptance criteria, one executing the next task, one running deterministic verification, and one updating documentation — before cycling back if anything fails.
